# Repository Guidelines

## Project Structure & Module Organization
This repo orchestrates end-to-end Playwright testing. Specs live under `tests/` (responsive, functionality, accessibility, visual) with helpers in `utils/`; suites share fixtures via `tests/unit` and `utils/test-fixtures.js`. Site manifests sit in `sites/` (`*-local.json`, `*-live.json`) and automation utilities reside in `scripts/`. Generated artifacts (`reports/`, `test-results/`) should remain untouched outside of cleanup scripts. Documentation and mock assets live in `docs/`, while `.github/workflows/` keeps the CI (Continuous Integration) glue. Update `README.md` whenever guidance changes and note the current commit hash in its footer so contributors know which revision the docs describe.

## Build, Test, and Development Commands
Run `npm run setup` after cloning to install dependencies and Playwright browsers. Execute `node run-tests.js --site=<name> --functionality` for the default Chrome desktop sweep; add `--responsive`, `--visual`, or `--accessibility`, or combine them with `--all-suites --exclude visual`. Use `--pages <n|all>` to cap manifests, `--project=<browser>` for Playwright projects, and `--workers=<count|auto>` to tune concurrency. `npm run test:unit` covers helper logic, and `npm run styles:build` recompiles the HTML reporter CSS after editing `styles/report/report-styles.scss`. Refresh manifests via `npm run discover -- <site|https://base.url>`, and prune artifacts with `npm run clean:reports`, `npm run clean:test-results`, and `npm run clean:manifests`.

## Coding Style & Naming Conventions
JavaScript stays in CommonJS with 2-space indentation, semicolons, and single quotes. Keep filenames descriptive (`responsive.layout.structure.spec.js`, `site-loader.js`) and prefer kebab-case for configs (`example-site.json`). Reuse helpers in `utils/` instead of reimplementing logic, add comments only for non-obvious flows, and avoid non-ASCII text unless already present. Run `npm run lint` before proposing changes, follow Conventional Commits (e.g., `fix(reports): guard empty suite summaries`), and ensure documentation mirrors behavior before opening a PR.
